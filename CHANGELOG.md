# Changelog

All notable changes to the Azure MCP Documentation Generator are documented here.

Format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

## [Unreleased]

### Added

- **Step 4 now always emits two per-namespace tool-family files ‚Äî one without CLI tabs and one with (`{namespace}-cli.md`)** ‚Äî The tool-family assembly step (`ToolFamilyCleanupStep`) previously overwrote the assembled article in place with CLI-tabbed content, leaving a single file. It now writes **two** files for every namespace via the new `CliVariantWriter` service: the canonical `tool-family/{namespace}.md` stays **plain MCP content with no CLI tabs** (never modified by the CLI step), and a sibling `tool-family/{namespace}-cli.md` receives the CLI tabs. When CLI tabs are disabled for a namespace (`cli-tab-config.json`) or no CLI data is available, the `-cli.md` file is written as an **exact copy** of the canonical article ‚Äî so exactly two files are always produced. Both the in-process reducer path and the subprocess-fallback path route through the same `ApplyCliTabWrappingAsync` ‚Üí `CliVariantWriter.WriteVariantsAsync` logic (the previously duplicated inline CLI-tab block was replaced with the shared call), and CLI-variant writes are non-fatal (surfaced as warnings) so they never fail the pipeline. Stale-file cleanup on rename now also removes the matching `{familyName}-cli.md`. 12 unit tests in `CliVariantWriterTests` cover variant path resolution (suffix placement, directory preservation, hyphenated/bare names), content branches (tabs injected when allowed+data; exact copy when disabled/no-data/null), and the write contract (canonical stays plain, `-cli.md` always written, missing-canonical returns null and writes nothing).
  - **Multi-namespace merge covers both variants:** the merge-group merge (`merge-namespaces.sh`) now merges the CLI-tab variant under the **same rules** as the canonical article ‚Äî for each merge group it merges `{member}-cli.md` ‚Üí `{primary}-cli.md` (primary frontmatter/overview/related + all members' tool sections in order + updated `tool_count`), preserving the `#### [Azure MCP CLI]`/`#### [MCP Server]` tab markers. The canonical merge is required (a missing member article skips the whole group); the `-cli.md` merge is best-effort and never blocks the canonical merge. A `NamespaceMergerTests` contract test locks that the typed `NamespaceMerger.Merge` applies identical rules to CLI-tab-styled content.

- **MCP tool annotation blocks now rendered as 3-row markdown tables** ‚Äî The `[Tool annotation hints](index.md#tool-annotations-for-azure-mcp-server):` section now emits a proper 3-row markdown table (header, separator, values) instead of the old single-line `Name: emoji | Name: emoji ...` inline format. Column order is fixed: Destructive, Idempotent, Open World, Read Only, Secret, Local Required. `true`‚Üí‚úÖ, `false`‚Üí‚ùå. This matches the published format in the Azure MCP docs (reference PR MicrosoftDocs/azure-dev-docs-pr#9391).
  - **`annotation-template.hbs`** and **`param-annotation-template.hbs`** updated to emit the table at Step 1.
  - **New `AnnotationTableFixer` service** (Step 4 post-processor) deterministically converts any remaining inline annotation lines to table format ‚Äî idempotent, so already-table input passes through unchanged.
  - **`ToolFamilyPostAssemblyValidator`** now blocks articles that contain the old inline format, and verifies that each annotation link has a matching table separator row.
  - **Regression guard tests** (`R-AP6`, `R-AP7` in `AnnotationPlacementRegressionTests`) **fail** if the old `Destructive: ‚ùå | ...` inline format appears in any assembled article.
  - **17 unit tests** (14 [Fact] + 1 [Theory] with 3 inline-data cases) in `AnnotationTableFixerTests` cover inline‚Üítable conversion, partial-field fail-safe (unconverted when fewer than 6 fields), idempotency, multi-block documents, and passthrough.

- **Post-processor now collapses multiple consecutive blank lines** ‚Äî `AcrolinxPostProcessor.Process` runs a final deterministic cleanup (`CollapseBlankLines`) that collapses any run of two or more consecutive blank lines (empty or whitespace-only) into a single blank line in the article body. It preserves the newline style (CRLF or LF) of each run's first line break and leaves single blank lines between blocks intact. This removes the stray double blank lines that LLM-synthesized sections and template conditionals could introduce. Six unit tests cover triple/many newlines, single-blank preservation, whitespace-only blanks, CRLF preservation, and end-to-end `Process` integration.
- **H1 paragraph now includes a one-sentence "when to use" summary** ‚Äî Each generated skill page appends a single sentence-based summary of when to reach for the skill to the intro paragraph under the H1, immediately after the AI-rewritten description (same paragraph). This summarizes ‚Äî and does not duplicate ‚Äî the detailed `## When to use this skill` section, which is retained in full. The summary is LLM-synthesized from the skill's `UseFor`/`DoNotUseFor` data via a new keyless `ILlmRewriter.SynthesizeWhenToUseSummaryAsync` (orchestrator step `llm-synthesize-when-to-use`), with a deterministic mechanical fallback (`SkillPageGenerator.BuildWhenToUseSummary`) when the LLM is unavailable. The frontmatter `description` is unchanged (intro only ‚Äî no summary appended). New `whenToUseSummary` parameter threads through `ISkillPageGenerator.Generate` ‚Üí `BuildContext` ‚Üí the Handlebars template (`{{#if whenToUseSummary}}`). Six unit tests cover LLM synthesis, NoOp/mechanical parity, BuildContext pass-through and fallback, the generic no-`UseFor` sentence, and the rendered H1-paragraph composition.
- **Generated-skills output folder now also stamped with a run timestamp** ‚Äî The output directory suffix now appends a `yyyy-MM-dd-HHmmss` run timestamp after the all-up Azure Skills version, so each run lands in a distinct folder (e.g. `../generated-skills/` ‚Üí `../generated-skills-1.1.72-2026-05-31-162525`), matching the timestamped convention of the other generator folders. `AzureSkillsVersionResolver.ApplyVersionSuffix` gained an optional `DateTimeOffset? timestamp` parameter; both `generate-skill` and `generate-skills` CLI handlers pass `DateTimeOffset.Now`. When the version is missing only the timestamp is appended; when both are absent the folder name is unchanged. Three new unit tests cover version+timestamp, timestamp-only, and neither.
- **Generated-skills output folder is now stamped with the all-up Azure Skills version** ‚Äî The output directory is suffixed with the overall `azure` plugin version read from `skills-source/plugin.json` (e.g. `--out ../generated-skills/` produces `../generated-skills-1.1.72/`). This is the package-wide Azure Skills version, NOT any individual skill's version. New `AzureSkillsVersionResolver` (`SkillsGen.Core/Versioning/`) resolves the version from `plugin.json` at the source path or its parent and applies the suffix; both `generate-skill` and `generate-skills` CLI handlers in `Program.cs` apply it after parsing `--source-path`. When `plugin.json` is missing or has no `version`, the folder name is left unchanged. 12 unit tests cover version resolution (present/parent/missing/malformed/null source) and suffix application (trailing slash/backslash/blank version).

### Fixed

- **`main` build-and-test gate restored to green (Tier 0)** ‚Äî the fast CI gate had been red for days due to 15 failing tests, blocking all merges (branch protection). Four independent causes were addressed: **(1)** the orphaned `DocGeneration.Steps.ToolFamilyCleanup.Validation.Tests` project (9 failures) still shelled out to `Validate-ToolFamily-PostAssembly.ps1`, which was deleted when post-assembly validation was ported to the C# `ToolFamilyPostAssemblyValidator` (whose own tests remain live); the dead project was removed from the solution. **(2)** `ParameterCoverageChecker` had an over-broad single-required-param fallback that marked any prompt containing a quoted/backtick/URL value as "covered" even when the parameter word was absent (1 false-negative guard-test failure); the fallback was removed, leaving the whitelisted name-like fallback intact. **(3)** Two `PreAiValidationGateObservabilityTests` used a hardcoded Windows path separator (`".\\generated-*"`) that stayed literal on Linux ‚Äî the runner normalizes separators via `PipelineContextFactory.NormalizePathSeparators` while the test's raw `Path.Combine` did not, so the runner wrote `metrics.json` to a different path than the test asserted (passed on Windows, failed on Linux CI); the tests now use plain relative paths. **(4)** The `build-and-test` workflow ran the entire solution with no category filter, so environment-dependent tests (`E2E`, `Smoke`, `SmokeTest`, `RequiresGeneration`) that require real generation output or a full pipeline run gated every PR; the fast gate now excludes those categories via `--filter`, and a tracking issue covers running them in a separate generation-dependent job (and re-tagging deterministic unit tests currently mislabeled `E2E`).
- **The `0x1A` CLI-metadata crash is now fixed at the source ‚Äî sanitize once on write, not on every read** ‚Äî the upstream Azure MCP CLI metadata output can contain raw control characters (notably `0x1A` SUB) inside JSON string values, which make `JsonDocument.Parse`/`JsonSerializer.Deserialize` throw `'0x1A' is invalid within a JSON string`. Earlier passes chased this by sanitizing at every downstream read site, but `McpCliMetadata` ‚Äî the **single writer** of `cli-version.json`, `cli-output.json`, and `cli-namespace.json` ‚Äî was still writing the **raw** upstream bytes to disk (it only sanitized an in-memory copy for its own namespace-mapping parse), so the artifacts on disk stayed dirty and every reader had to defend itself. The fix now sanitizes **once, at write time**, in the only place raw CLI output enters the system: a new `CliMetadataWriter.WriteArtifactsAsync` routes all three artifacts through `Shared.JsonControlCharacterSanitizer.StripInvalidControlCharacters` before they touch disk, guaranteeing valid JSON downstream. Because the source artifacts are now clean (and the per-namespace filtered/enriched files are re-serialized through `System.Text.Json`, which escapes control chars), the read-time sanitization added in prior passes was **removed** from all reader sites (`CliJsonMapper`, `CliVersionReader`, `CliMetadataLoader`, `BrandMappings`, `RawToolGeneratorService`, `Annotations.DocumentationGenerator`, `ExamplePrompts` generation/validation, `HorizontalArticleGenerator`, `ArticleOutlineBuilder`, `ToolMetadataEnricher`) ‚Äî readers now trust the source-cleaned artifacts. The shared `JsonControlCharacterSanitizer` helper and its tests remain (now used by the writer). New `CliMetadataWriterTests` lock the contract: raw `0x1A` in the tools and namespace JSON is stripped so the on-disk files parse cleanly, and tool data is preserved (verified RED when `Sanitize` is neutered to a pass-through).
- **Added the "Azure Resilience" brand-to-server mapping** ‚Äî `data/brand-to-server-mapping.json` now maps the Azure Resilience brand so the resilience namespace resolves to its MCP server during Step 0 bootstrap.

- **Brand-mapping validator (Step 0) no longer requires `FOUNDRY_API_KEY` under keyless auth** (PR #702) ‚Äî the `DocGeneration.Steps.Bootstrap.BrandMappings` entry point invokes AI generation (`GenerativeAIClient`) to suggest brand mappings, but its inline option validation unconditionally listed `FOUNDRY_API_KEY` as required. Under keyless auth (`FOUNDRY_USE_DEFAULT_CREDENTIAL=true`, no API key) it therefore skipped AI generation and emitted placeholder mappings (exit code 2) even with a valid endpoint and deployment. The validation is now extracted to a testable `Program.ValidateAIOptions` that requires the API key only when default credential is not enabled ‚Äî the fourth and final AI entry point brought in line with `AiCapabilityProbe`, `GenerativeAIClient`, and the `ToolFamilyCleanup`/`HorizontalArticleProgram` validators. Two `ValidateAIOptionsKeylessTests` prove the API key is optional under default credential and still required without it.
- **Bootstrap AI probe `.env` parsing is now byte-for-byte consistent with the runtime AI client** (PR #702) ‚Äî `AiCapabilityProbe` duplicated a simpler `.env` parser that (a) stripped only double quotes, so a single-quoted flag (`FOUNDRY_USE_DEFAULT_CREDENTIAL='true'`) was read as the literal `'true'` and treated as false, wrongly requiring an API key, and (b) merged process-env first then let `.env` overwrite it, disagreeing with `GenerativeAIOptions` (which treats the process environment as authoritative) when the same variable was set differently in each. The probe now delegates entirely to `GenerativeAIOptions.LoadFromEnvironmentOrDotEnv(mcpToolsRoot)`, eliminating the duplicated logic so quote stripping (both single and double) and process-env-vs-`.env` precedence can never diverge from the runtime client. Two new `AiCapabilityProbeTests` cover the single-quoted flag and a process-env-vs-`.env` conflict resolving identically to the shared loader.
- **`ToolFamilyCleanup` missing-variable error now names the correct env var (`FOUNDRY_MODEL_NAME`)** (PR #702) ‚Äî when the deployment/model name was missing, `Program.ValidateAIOptions` and the usage help reported `TOOL_FAMILY_CLEANUP_FOUNDRY_MODEL_NAME`, but the deployment is loaded from `FOUNDRY_MODEL_NAME` via `GenerativeAIOptions.LoadFromEnvironmentOrDotEnv` (no code promotes `TOOL_FAMILY_CLEANUP_FOUNDRY_MODEL_NAME` to it), so following the guidance did not resolve the error. Both the error string and the usage text now report `FOUNDRY_MODEL_NAME`, matching `HorizontalArticleProgram`. A `ValidateAIOptionsModelNameTests` regression asserts the reported variable name. `AiCapabilityProbe.ProbeAsync` hardcoded `FOUNDRY_API_KEY`, `FOUNDRY_ENDPOINT`, and `FOUNDRY_MODEL_NAME` as always-required, so any `.env` configured for default-credential auth (`FOUNDRY_USE_DEFAULT_CREDENTIAL=true` with no `FOUNDRY_API_KEY`, as written by `populate-env.ps1 -UseDefaultCredential`) failed Step 0 bootstrap for every namespace with exit code 1. The probe now reads `FOUNDRY_USE_DEFAULT_CREDENTIAL` from the same merged process-env + `.env` sources and, when truthy (`1`/`true`/`yes`/`on`, case-insensitive ‚Äî mirroring `GenerativeAIOptions.ParseBoolean`), requires only `FOUNDRY_ENDPOINT` and `FOUNDRY_MODEL_NAME`; the API key stays required when default credential is not enabled. This aligns the probe with the runtime AI client (`GenerativeAIClient`), which already supported `DefaultAzureCredential`. New `AiCapabilityProbeTests` cover the default-credential-without-key regression, truthy flag variants, the API-key happy path, and the no-key/no-flag and explicitly-false guards.
- **Tool-family reducer (Step 4) and horizontal article generation (Step 6) now resolve keyless AI config from the mcp-tools root** ‚Äî `ToolFamilyCleanupStep.ResolveFamilyCleanupAsync` built its `CleanupGenerator` with an empty `new GenerativeAIOptions()`, and `HorizontalArticlesStep` loaded `GenerativeAIOptions.LoadFromEnvironmentOrDotEnv()` with no base path. Both steps run with an isolated working directory, so `FOUNDRY_ENDPOINT` / `FOUNDRY_MODEL_NAME` / `FOUNDRY_USE_DEFAULT_CREDENTIAL` from `.env` were never resolved and `GenerativeAIClient` threw `Azure OpenAI configuration incomplete` ‚Äî breaking the keyless reducer path (Step 4) and forcing a subprocess fallback that also failed keyless. Both now resolve options through a `ResolveGenerativeAIOptions(context.McpToolsRoot)` seam that anchors `.env` discovery to the mcp-tools root, making keyless (`DefaultAzureCredential`) the fully-supported, intended auth path with no API key required. Two `ReducerAiOptionsTests` prove each step resolves the endpoint, deployment, and default-credential flag from a temp keyless `.env`.
- **Tool-family cleanup and horizontal-article subprocess entry points no longer require FOUNDRY_API_KEY under keyless auth** ‚Äî the standalone `ToolFamilyCleanup` (`Program.ValidateAIOptions`) and horizontal-article (`HorizontalArticleProgram.ValidateAIOptions`) executables unconditionally listed `FOUNDRY_API_KEY` as a required environment variable, so the Step 6 subprocess (and the Step 4 subprocess fallback) failed keyless with `Missing required environment variables (after .env fallback): FOUNDRY_API_KEY` even though the endpoint and deployment were present. Both now require the API key only when `FOUNDRY_USE_DEFAULT_CREDENTIAL` is not enabled, matching `AiCapabilityProbe` and `GenerativeAIClient` and making keyless (`DefaultAzureCredential`) the fully-supported, intended path across every pipeline entry point. Two `ValidateAIOptionsKeylessTests` in each subprocess test project prove the API key is optional under default credential and still required without it.
- **CLI parameter table display names no longer include leading switch prefixes** ‚Äî CLI-tab parameter table generation now strips one leading `--` or `-` from the rendered parameter name cell only (for example, `` `resource-group` `` instead of `` `--resource-group` ``). Raw `CliSwitch.Name` values are still retained for CLI examples, metadata matching, and validation. The fallback inline CLI parameter table uses the same shared display-name formatter, preventing generated Azure MCP tool articles from reintroducing leading `--` in parameter table name cells.
- **Derived display names now use brand-correct Azure product spelling** ‚Äî `SkillMarkdownParser.DeriveDisplayName` mechanically title-cased each hyphen-split slug token, which mangled mashed product tokens (e.g. `python-appservice-deploy` ‚Üí "Python Appservice Deploy"). It now consults a deterministic `ProductNameMap` per token *before* the acronym/title-case rules, substituting canonical product fragments (`appservice` ‚Üí "App Service", `cosmosdb` ‚Üí "Cosmos DB", `keyvault` ‚Üí "Key Vault", `appinsights` ‚Üí "Application Insights", `postgresql` ‚Üí "PostgreSQL", `openai` ‚Üí "OpenAI", etc.). This fixes both the H1 (`# Azure skill for {{displayName}}`) and the frontmatter `title` for SEO and brand correctness. Display names supplied directly via a source `display_name` frontmatter field are still used verbatim ‚Äî only slug-derived names are normalized. `DeriveDisplayName` is now `internal static` (testable via `InternalsVisibleTo`); five new theory cases cover the product map alongside the existing acronym/title-case cases.
- **Reverted lift-and-shift of skill frontmatter `description`** ‚Äî The `preserve-frontmatter-description` change (commit `7ed4d7b`) that copied the source `SKILL.md` frontmatter `description` verbatim has been reverted. The orchestrator again runs the `llm-rewrite-intro` step so the generated frontmatter `description` and the intro paragraph under the H1 are AI-rewritten for Microsoft voice (not a verbatim lift from source), and `AcrolinxPostProcessor` again applies static replacements to the frontmatter description. The generated H1/`title` continues to come from the template (`Azure skill for {{displayName}}`) and was never a source lift. The keyless `SynthesizeWhatItProvides` LLM call is unaffected.
- **Skills generation rewriter now works with gpt-5 / o-series reasoning models** ‚Äî Two HTTP 400/empty-content failures during keyless generation against `gpt-5-mini` are fixed in `AzureOpenAiRewriter`: (1) reasoning models reject an explicit `temperature` (`unsupported_value` ‚Äî only the default `1` is allowed), so the rewriter now omits `Temperature` for the `gpt-5`/`o1`/`o3`/`o4` families and only pins `0.3` for other models; (2) those same models spend output tokens on internal reasoning, so the previous `MaxOutputTokens = 500` cap left no room for visible content (Azure returned empty content). The budget is now model-aware ‚Äî `4000` for reasoning families, `500` otherwise ‚Äî via new `SupportsCustomTemperature` and `MaxOutputTokensFor` helpers. Four unit tests cover both behaviours across `gpt-5-mini` and `gpt-4o`.
- **Skills generation keyless rewriter docs and retry coverage** ‚Äî Added a regression test proving a 429 from the injected `IChatClient` retries and succeeds, and removed the unused `FOUNDRY_USE_DEFAULT_CREDENTIAL` setting from the skills-generation README.
- **Skills LLM rewrite restored ‚Äî keyless auth and SDK path fix** ‚Äî `AzureOpenAiRewriter` now goes through the `Microsoft.Extensions.AI` `IChatClient` abstraction (`AsIChatClient`) instead of the raw `Azure.AI.OpenAI` `ChatClient`. This fixes the `MissingMethodException: get_SerializedAdditionalRawData()` that was thrown before any network call (binary mismatch between `Azure.AI.OpenAI` 2.1.0 and the transitively-resolved `OpenAI` 2.9.1), which caused every AI-enhanced section to silently fall back to mechanical text. The rewriter's primary constructor now accepts an injected `IChatClient`, making it unit-testable. New tests in `AzureOpenAiRewriterTests` cover intro/knowledge/what-it-provides/workflow-step rewriting against a stubbed `IChatClient`.

### Changed

- **Default `ms.reviewer` no longer hardcodes `mbaldwin`** ‚Äî Tool-family cleanup now sets `MetadataConstants.Reviewer = ""` and emits a blank `ms.reviewer:` line without trailing whitespace when the reviewer value is empty. The horizontal article template and committed examples now match the empty-value frontmatter behavior so generated and sample output keep the `ms.reviewer` key without substituting another alias.
- **Skills generation uses keyless authentication only** ‚Äî this repository never uses API keys. `AzureOpenAiRewriter.CreateKeyless(endpoint, model, ...)` builds a `DefaultAzureCredential`-backed chat client (managed identity / `az login`), and `Program.cs` now wires the rewriter from `FOUNDRY_ENDPOINT` + `FOUNDRY_MODEL_NAME` alone. The `FOUNDRY_API_KEY` code path and `AzureKeyCredential` usage have been removed from the skills pipeline. `Azure.Identity` is now a direct reference of `SkillsGen.Core`.
- **Block internal MCP tool names and `### Related tools` sections from skills output** (#684) ‚Äî The LLM could inject `### Related tools` sections with `Azure__*` tool call names into generated article bodies. Two defences added: (1) `AcrolinxPostProcessor` now strips any `### Related tools` section (heading + body until the next heading or EOF) and any lines containing `Azure__` patterns before other processing runs; (2) `SkillPageValidator` now raises a POLICY error if either pattern survives post-processing. This prevents implementation details from reaching customer-facing documentation.
- **`start.sh` now copies the repo-root `.env` into `mcp-tools/.env` before running the pipeline** ‚Äî the script now fails immediately with a clear message when `.env` is missing at the repo root, instead of letting credential validation fail later in bootstrap.
- **Step 6 prompt paths now resolve correctly when running in-process via the reducer** ‚Äî `HorizontalArticleGenerator` previously resolved prompt and template file paths using `Path.GetFullPath` against the current working directory. When run as a subprocess the CWD was correctly set to `mcp-tools/`, but when run in-process (reducer path) the CWD was the repo root, causing a `FileNotFoundException` for every prompt file. The constructor now accepts an optional `mcpToolsRoot` parameter; when supplied, all paths are anchored to that directory instead of the CWD. `HorizontalArticlesStep.ResolveArticleRendererAsync` passes `context.McpToolsRoot` so the reducer path always finds the prompts. The fallback (null `mcpToolsRoot`) retains the old CWD-based behaviour for the subprocess path. Four unit tests added: `Constructor_WithMcpToolsRoot_ResolvesPromptPathRelativeToMcpToolsRoot`, `Constructor_WithMcpToolsRoot_ResolvesTemplatePathRelativeToMcpToolsRoot`, `Constructor_WithMcpToolsRoot_AllPromptFilesResolveUnderMcpToolsRoot`, and `Constructor_WithoutMcpToolsRoot_FallsBackToCwdForPromptPath`.

- **BootstrapStep no longer fails when a newer prerelease of `azure.mcp` is already installed** ‚Äî `mcp-tool-version.txt` at repo root is now the single source of truth for the azure.mcp version. `BootstrapStep` reads this file and passes `--version` to `dotnet tool update`, preventing a downgrade from `3.0.0-beta.15` to the latest stable `2.0.2`. "Already at this version" exit codes are treated as success. `run-focus.sh` also reads from the same file. When the file is absent, the previous behaviour (update to latest) is preserved with a warning.

- **Cosmos regeneration no longer downgrades `azure.mcp` from prerelease to stable** ‚Äî `run-focus.sh` now pre-installs `azure.mcp@3.0.0-beta.15` before dispatching any namespace and always forwards `--skip-npm-update` to `start.sh`, preventing `BootstrapStep` from attempting a conflicting stable-tool downgrade during focus runs.

- **#673: MCP Server tab now appears before CLI tab** ‚Äî `CliTabWrapper.BuildTabBlock()` previously emitted the Azure MCP CLI tab first, then the MCP Server tab. The order is now corrected: MCP Server tab is emitted first, followed by the Azure MCP CLI tab. All existing `CliTabWrapperTests` updated to reflect the new ordering; new unit test `WrapWithTabs_McpServerTabAppearsBeforeCliTab` added. Resolves #673.

- **#664A: `prompt-preview.txt` never written for AI/Hybrid pipeline steps** ‚Äî `ObservabilityWriter.WritePromptPreview()` added and called for all non-deterministic steps (Steps 2, 3, 4, 6) in `PipelineRunner.WriteObservabilityOutputs()`. Previously only deterministic steps wrote `prompt-preview-na.txt`, leaving AI/Hybrid steps without the file `StageOutputContract` expected, causing a spurious "observability contract missing file" warning on every AI step run. The new method writes a pipeline-level placeholder (`"AI step ‚Äî prompt preview not captured at pipeline level."`) that satisfies the contract. Test `RunAsync_AiStep_MissingPromptPreviewLogsWarning` renamed to `RunAsync_AiStep_WritesPromptPreviewAndNoContractWarning` and updated to assert the file is present and no contract warning is emitted; `WritePromptPreview_CreatesFileWithExpectedContent` unit test added. Resolves #664 sub-issue A.
