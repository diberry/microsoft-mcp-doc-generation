# Azure MCP Server Tool Coverage Audit

Deterministic (script-based) coverage audit that compares published Azure MCP Server tool articles against `tools-list.json`. Identifies gaps in documentation coverage without requiring LLM inference.

## Relationship to Other Skills

| Skill | Purpose | Method |
|-------|---------|--------|
| **This skill** (`mcp-coverage-audit`) | "What tools/params/annotations are missing from docs?" | Deterministic regex + JSON comparison |
| `azure-ai-tools-validation` | "Is the generated content correct?" | AI-driven Phase 1 + Phase 2 checks |
| `azure-ai-tools-mcp` | "How do I regenerate content for a new release?" | Workflow orchestration |

**Use this skill when:** You want a fast, repeatable, scriptable answer to "is everything documented?" without running the generation pipeline or invoking AI.

## What It Detects

| Check | How | Deterministic? |
|-------|-----|----------------|
| **Undocumented tools** | Regex scan for `<!-- {command} -->` or `<!-- @mcpcli {command} -->` markers in articles | ✅ Yes |
| **Missing articles** | Namespace→filename mapping + file existence check | ✅ Yes |
| **Missing parameters** | Parse markdown parameter tables, compare against `option[]` in JSON | ✅ Yes |
| **Annotation mismatches** | Parse `Destructive: ❌ | ...` lines, compare against `metadata{}` in JSON | ✅ Yes |
| **Common param leakage** | Flag tools documenting `--learn`, `--tenant`, etc. per-tool | ✅ Yes |
| **Open PR coverage** | Scans PR branches via `git show` to distinguish "truly missing" from "covered by open PR" | ✅ Yes |

## Prerequisites

| Requirement | Details |
|---|---|
| `tools-list.json` | Latest version from `test-npm-azure-mcp/{version}/tools-list.json` in the doc-generation repo |

## Scripts

### `Get-McpOpenPRs.ps1` — Auto-discover open PRs

Queries GitHub via `gh pr list` for open PRs that touch `articles/azure-mcp-server/tools/`. Generates a datetime-stamped JSON file compatible with `Scan-McpToolCoverage.ps1 -OpenPRsJson`.

```powershell
# Generate open PRs JSON (uses authenticated gh CLI user by default)
.\scripts\Get-McpOpenPRs.ps1 `
  -OutputDir "C:\project-dina-ai-dev-tools\projects\azure-ai-tools\status"

# Custom repo/author
.\scripts\Get-McpOpenPRs.ps1 `
  -OutputDir "C:\project-dina-ai-dev-tools\projects\azure-ai-tools\status" `
  -Repo "MicrosoftDocs/azure-dev-docs-pr" `
  -Author "diberry"
```

**Output:** `open-prs-{yyyy-MM-dd-HHmmss}.json` — same format consumed by `Scan-McpToolCoverage.ps1 -OpenPRsJson`.

### `Scan-McpToolCoverage.ps1` — Full coverage audit

Scans all articles against all tools in `tools-list.json`. Outputs both console table and optional JSON report.

```powershell
# Full audit (all namespaces) with report
.\scripts\Scan-McpToolCoverage.ps1 `
  -ToolsJsonPath "C:\project-dina-ai-dev-tools\repos\public-diberry-microsoft-mcp-doc-generation\test-npm-azure-mcp\3.0.0-beta.10+7287903f962dd029489594e2ae68842f3e10ac30\tools-list.json" `
  -ArticlesDir "C:\project-dina-ai-dev-tools\repos\public-microsoftdocs-azure-dev-docs-pr\articles\azure-mcp-server\tools" `
  -ReportDir "C:\project-dina-ai-dev-tools\projects\azure-ai-tools\status" `
  -OutputJson "coverage-report.json"

# Two-step workflow: generate open PRs then audit with PR awareness
$prsFile = .\scripts\Get-McpOpenPRs.ps1 `
  -OutputDir "C:\project-dina-ai-dev-tools\projects\azure-ai-tools\status"

.\scripts\Scan-McpToolCoverage.ps1 `
  -ToolsJsonPath "..." `
  -ArticlesDir "..." `
  -OpenPRsJson $prsFile `
  -RepoRoot "C:\project-dina-ai-dev-tools\repos\public-microsoftdocs-azure-dev-docs-pr" `
  -ReportDir "C:\project-dina-ai-dev-tools\projects\azure-ai-tools\status"

# Single namespace
.\scripts\Scan-McpToolCoverage.ps1 `
  -ToolsJsonPath "..." `
  -ArticlesDir "..." `
  -Namespace "deploy"
```

**Open PRs JSON format** (auto-generated by `Get-McpOpenPRs.ps1`):
```json
[
  {"pr": 9015, "namespace": "azurebackup", "branch": "diberry/mcp-tool-backup", "file": "azure-backup.md"},
  {"pr": 9019, "namespace": "azureterraform", "branch": "diberry/mcp-tool-terraform", "file": "azure-terraform.md"},
  {"pr": 9047, "namespace": "deviceregistry", "branch": "diberry/mcp-tool-deviceregistry", "file": "azure-device-registry.md"}
]
```

**Output:**
- **Console:** Color-coded summary with missing tools, params, and annotation mismatches (split by main vs open PRs)
- **`-ReportDir`:** Saves actionable markdown report (`mcp-coverage-report-{date}.md`) with issue tables, PR coverage section
- **`-OutputJson`:** Full structured JSON for CI integration (includes `tools_in_open_prs` counts)

## Detection Patterns

### Tool marker (identifies documented tools)

```
<!-- @mcpcli {namespace} {action} [{subaction}] -->
```

Examples:
- `<!-- @mcpcli deploy app logs get -->`
- `<!-- @mcpcli storage account get -->`
- `<!-- @mcpcli compute vm create -->`

Each marker = one documented tool. The `command` field in `tools-list.json` matches the marker content.

### Parameter Exclusion Rules

The script uses the same rules as published articles:

1. **`--learn`, `--subscription`, `--tenant`, `--auth-method`, `--retry-*`** — ALWAYS excluded. Global parameters, never documented per-tool.
2. **`--resource-group`** — Excluded UNLESS `"required": true` for that specific tool. Many tools require this (e.g., `compute vm create`), so it will appear as expected when required.

This matches the article authoring rule: common/global params are not in individual tool parameter tables unless they are required for that tool.

### Parameter table (identifies documented parameters)

```markdown
| Parameter | Required or optional | Description |
|-----------|---------------------|-------------|
| **Param Name** | Required | Description... |
```

Bold text in column 1 = parameter name. Compared against `option[].name` (minus `--` prefix, kebab→title case).

### Parameter Fuzzy Matching

Articles sometimes use natural language variations of parameter names (e.g., `--account` documented as "account name"). The script uses a two-pass approach:

1. **Exact match** — direct comparison of kebab-to-display-name and camelCase variants
2. **Fuzzy match** — for unmatched params, checks unmatched doc table entries using:
   - Substring containment (doc param contains JSON param or vice versa)
   - Word overlap (≥50% shared words between the two names)

This reduces false positives where the parameter IS documented but with a slightly different display name.

### Annotation line (identifies documented metadata)

```
Destructive: ❌ | Idempotent: ✅ | Open World: ❌ | Read Only: ✅ | Secret: ❌ | Local Required: ✅
```

Compared against `metadata.{key}.value` (boolean → ✅/❌).

## Namespace → Filename Mapping

Not all namespaces map to `azure-{namespace}.md`. Key exceptions:

| CLI Namespace | Article Filename |
|---|---|
| `appconfig` | `app-configuration.md` |
| `applicationinsights` | `application-insights.md` |
| `group` | `resource-group.md` |
| `speech` | `ai-services-speech.md` |
| `kusto` | `azure-data-explorer.md` |
| `datadog` | `azure-native-isv.md` |
| `foundryextensions` | `azure-foundry.md` |
| `functionapp` | `azure-functions.md` |
| `get` | `azure-best-practices.md` |

The full mapping is maintained in the script. When a filename isn't in the map, the script falls back to scanning all `.md` files for tool markers matching the namespace.

## Integration

### With `azure-ai-tools-validation`

This skill provides the **data** (what's missing). The validation skill provides the **rules** (what should be done about it). They are complementary:

| Skill | Question Answered | When to Use |
|-------|-------------------|-------------|
| `azure-ai-tools-mcp-coverage-audit` (this) | "What tools/params/annotations are missing?" | Before PRs, gap analysis, CI checks |
| `azure-ai-tools-validation` | "Is the content structurally and factually correct?" | After generation, before merge |

Use together:
1. Run `Scan-McpToolCoverage.ps1` → get gap report (main + open PRs)
2. Feed gaps into `azure-ai-tools-validation` Phase 2 for detailed accuracy checks on documented tools
3. Use gaps to prioritize which PRs to create/update

### With pipeline decision gate

The coverage audit can serve as input to the pipeline decision gate (issue #571): if a namespace has undocumented tools, it's a candidate for a PR regardless of CHANGELOG.

### In CI/PR review

Run the script as a PR check to verify a PR doesn't reduce coverage:
- Before merge: documented tool count ≥ current main branch count
- New tools in JSON must have corresponding tool markers (HTML comments) in the PR

## Limitations

- Does NOT validate prose quality or description accuracy (use `azure-ai-tools-validation` Phase 2)
- Does NOT check article ordering or style
- Parameter matching uses heuristic name normalization — may have false positives on unusual parameter names
- Annotation parsing assumes the standard `Key: ✅/❌ | Key: ✅/❌` format
